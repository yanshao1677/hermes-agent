### 一、AIAgent 是怎么创建的

AIAgent 的 `__init__` 在 `run_agent.py:512` 开始，是一个非常庞大的构造函数。核心初始化流程可以拆成 5 个阶段：

**1. 运行时模式与客户端选择（512-941）**

AIAgent 根据传入的 `provider`、`base_url`、`api_mode` 参数，自动判断使用哪种 API 模式：

```642:658:run_agent.py
if api_mode in {"chat_completions", "codex_responses", "anthropic_messages"}:
    self.api_mode = api_mode
elif self.provider == "openai-codex":
    self.api_mode = "codex_responses"
elif (provider_name is None) and "chatgpt.com/backend-api/codex" in self._base_url_lower:
    self.api_mode = "codex_responses"
    self.provider = "openai-codex"
elif self.provider == "anthropic" or (provider_name is None and "api.anthropic.com" in self._base_url_lower):
    self.api_mode = "anthropic_messages"
    self.provider = "anthropic"
```

三种 API 模式分别对应：
- **chat_completions**：标准 OpenAI `/v1/chat/completions` 协议（最通用）
- **codex_responses**：OpenAI Responses API（GPT-5、Codex 系列需要）
- **anthropic_messages**：Anthropic 原生 Messages API（支持 prompt caching）

客户端初始化时，Anthropic 模式用 `build_anthropic_client()`，其他模式用 OpenAI SDK，还会根据 URL 自动注入特定 header（OpenRouter、Qwen Portal、Copilot 等）。

**2. 工具发现与 toolset 过滤（960-1070）**

调用 `get_tool_definitions()` 获取所有可用工具 schema，`check_toolset_requirements()` 检查每个 toolset 的依赖（API key 等）是否满足，最终得到 `self.valid_tool_names`——当前实际可用的工具名集合。这决定了后面 system prompt 里注入哪些行为指导。

**3. 记忆系统初始化（1082-1139）**

先加载内建 `MemoryStore`（MEMORY.md + USER.md），然后看 `memory.provider` 配置是否要加载外部记忆提供商（如 Honcho）。关键：只有 `memory_enabled=True` 或 `user_profile_enabled=True` 时，`MemoryStore` 才会被创建并从磁盘加载。

**4. 上下文压缩器初始化**

`ContextCompressor` 从配置读取阈值、保护头尾条数等参数。压缩不是每轮都做，只在上下文逼近模型 context_length 时触发。

**5. 各种状态追踪器**

`_turns_since_memory`、`_iters_since_skill`（nudge 计数器）、`_user_turn_count`、`_cached_system_prompt`、迭代预算 `IterationBudget` 等。

入口在 `run_agent.py:7478`。核心流程：

**Step 1：初始化消息列表与状态重置（7581-7617）**

**Step 2：nudge 计数器检查（7604-7611）**

每轮进入时 `_turns_since_memory += 1`，达到 `nudge_interval`（默认 10）则标记 `_should_review_memory = True`，等本轮结束后触发后台审查。

**Step 3：system prompt 缓存与构建（7633-7672）**

这是阶段二最核心的逻辑：

```7633:7649:run_agent.py

    stored_prompt = None
    if conversation_history and self._session_db:

        if session_row:

    if stored_prompt:
        # 继续会话——复用旧的 system prompt，保持 Anthropic prefix cache 命中

```

注意：**Gateway 模式每条消息创建新的 AIAgent 实例**，所以它依赖 SQLite 存储的旧 prompt 来恢复缓存。如果每次都重建，Anthropic 的 prefix cache 就会失效，每次请求都要重新计算整个 system prompt 的 token。

**Step 4：主循环（while api_call_count < max_iterations）**

核心循环大致结构：

```
while 未超迭代预算:
    1. 构建 api_messages（注入 ephemeral context 到当前用户消息）
    2. 加上 system prompt 作为第一条消息
    3. 调用 LLM API
    4. 如果响应有 tool_calls:
         - 修复/校验工具调用（孤立 result、无效名称、无效 JSON）
         - 执行工具调用
         - 将工具结果追加到 messages
         - 继续循环
    5. 如果响应是纯文本（无 tool_calls）:
         - 这就是最终回答
         - break
```

**Step 5：轮次结束后的清理（10230-10262）**

- 检查 skill nudge 计数器
- 外部记忆提供商 `sync_all()`（同步本轮对话）
- 如果有 `_should_review_memory` 或 `_should_review_skills`，启动后台审查线程

### 三、`_build_system_prompt()` 组装了哪些层

这是阶段二最需要搞懂的方法，位于 `run_agent.py:3001`。它组装 **7 层**内容，按顺序拼接：

| 层 | 来源 | 说明 |
|----|------|------|
| **1. 代理身份** | `SOUL.md` 或 `DEFAULT_AGENT_IDENTITY` | 优先读 `~/.hermes/SOUL.md`（用户自定义人格），不存在则用硬编码的默认身份 |
| **2. 工具行为指导** | `MEMORY_GUIDANCE`、`SESSION_SEARCH_GUIDANCE`、`SKILLS_GUIDANCE` | 按可用工具名条件注入，告诉模型"该用就用" |
| **3. 用户/system_message** | 参数传入 | CLI 或 Gateway 可以传入额外的系统指令 |
| **4. 持久记忆（冻结快照）** | `MemoryStore.format_for_system_prompt()` | MEMORY.md 和 USER.md 的**构建时快照**，不是实时状态 |
| **5. 技能索引** | `build_skills_system_prompt()` | 只放技能名+描述的索引，全文需要时用 `skill_view` 加载 |
| **6. 项目上下文文件** | `.hermes.md` > `AGENTS.md` > `CLAUDE.md` > `.cursorrules` | 优先级：先找到谁就用谁（只取一个），加上 SOUL.md |
| **7. 时间戳+平台提示** | `PLATFORM_HINTS[platform]` | 当前时间、session ID、平台格式规则（如 WhatsApp 不渲染 markdown） |

**关键设计决策：**

层 4（记忆）用的是**冻结快照**，不是实时状态。`format_for_system_prompt()` 返回的是 `load_from_disk()` 时捕获的 `_system_prompt_snapshot`，中途写记忆不会更新它。这是为了保持 system prompt 稳定，让 Anthropic 的 prefix cache 始终命中。

### 四、`_cached_system_prompt` 为什么不能每轮都重建

这是阶段二最核心的"为什么"问题。答案涉及两个原因：

**原因一：Anthropic Prompt Caching**

`agent/prompt_caching.py` 实现了 **"system_and_3"策略**：在 Anthropic API 的消息中放置最多 4 个 `cache_control` 断点：

```41:72:agent/prompt_caching.py
def apply_anthropic_cache_control(
    api_messages: List[Dict[str, Any]],
    cache_ttl: str = "5m",
    native_anthropic: bool = False,
) -> List[Dict[str, Any]]:
    """Apply system_and_3 caching strategy to messages for Anthropic models.

    Places up to 4 cache_control breakpoints: system prompt + last 3 non-system messages.
    """
    # ...
    if messages[0].get("role") == "system":
        _apply_cache_marker(messages[0], marker, native_anthropic=native_anthropic)
        breakpoints_used += 1

    remaining = 4 - breakpoints_used
    non_sys = [i for i in range(len(messages)) if messages[i].get("role") != "system"]
    for idx in non_sys[-remaining:]:
        _apply_cache_marker(messages[idx], marker, native_anthropic=native_anthropic)
```

4 个断点分别是：system prompt + 最后 3 条非 system 消息。Anthropic 会缓存从消息开头到每个断点的内容，下次请求如果前缀相同，直接命中缓存，**节省约 75% 的输入 token 成本**。

如果每轮都重建 system prompt，哪怕只改了一个字符，整个 system prompt 的缓存就全部失效——对长会话来说代价巨大。

**原因二：模型行为稳定性**

system prompt 里包含了代理身份、记忆、技能索引、平台规则等。如果每轮都重建，可能因为配置变化、文件变化、记忆被写入等原因导致内容漂移，模型的身份认知和行为规则会在会话中途改变——用户会感受到模型"变了一个性格"。

**唯一重建的时机：上下文压缩后**（`_invalidate_system_prompt`）：

```3322:3331:run_agent.py
def _invalidate_system_prompt(self):
    """使缓存的系统提示词无效，强制在下一轮次重建。
    
    在上下文压缩事件后调用。还会从磁盘重新加载记忆，
    以便重建的提示词捕获此会话中的任何写入。
    """
    self._cached_system_prompt = None
    if self._memory_store:
        self._memory_store.load_from_disk()
```

压缩后旧消息被总结，新的 system prompt 会包含最新的记忆快照。此时 prefix cache 必然失效（消息列表变了），所以顺便重建 system prompt 没有额外损失。

### 五、`ephemeral_system_prompt` 是什么，为什么不进持久 system prompt

`ephemeral_system_prompt` 是 `AIAgent.__init__` 的一个参数（`run_agent.py:530`），用于临时性的、不希望持久化的指令。

它**不**出现在 `_build_system_prompt()` 中（注释明确说了：`run_agent.py:3078`），而是在构建 API 调用时才注入：

```7869:7885:run_agent.py
# Inject ephemeral context into the current turn's user message.
# Sources: memory manager prefetch + plugin pre_llm_call hooks
if idx == current_turn_user_idx and msg.get("role") == "user":
    _injections = []
    if _ext_prefetch_cache:
        _fenced = build_memory_context_block(_ext_prefetch_cache)
        if _fenced:
            _injections.append(_fenced)
    if _plugin_user_context:
        _injections.append(_plugin_user_context)
```

类似地，外部记忆提供商的 `prefetch()` 结果也是以 `<memory-context>` 标签注入到当前用户消息中，而不是放进 system prompt。

**为什么这样设计：**

1. **不破坏 system prompt 缓存**——ephemeral 内容每轮都可能变，如果放进 system prompt，每轮都导致缓存失效
2. **不污染会话持久化**——这些内容不会被保存到 SQLite session DB，不会被后续会话回放
3. **语义更准确**——prefetch 的记忆是"当前任务相关的背景信息"，不是"代理的永久规则"，放在用户消息层更合理

**核心设计哲学：** Hermes 把信息分成"什么必须稳定"（system prompt，整个会话不变）和"什么可以临时"（ephemeral、prefetch，每轮注入），前者保证缓存和行为一致性，后者保证当前任务有足够的上下文。两者不混在一起，是 Hermes 区别于简单"每轮拼大 prompt"方案的关键。
