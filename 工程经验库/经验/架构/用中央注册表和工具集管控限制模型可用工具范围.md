# 用中央注册表和工具集管控限制模型可用工具范围

## 1. 背景与场景

工具型智能体往往需要几十种能力：文件、终端、网络、浏览器、记忆、委托、平台消息、外部插件。若直接把所有工具暴露给模型，模型会看到过多无关能力，也会调用当前环境不可用或当前任务不应使用的能力。

这个决策适用于工具数量持续增长、工具有可选依赖、不同运行入口需要不同能力集的智能体系统。

## 2. 核心问题

要解决的问题不是“如何写一个工具函数”，而是“如何让模型在每轮只看见可用且被允许的工具”。关键约束包括：

- 工具规格、执行函数和可用性判断必须保持一致。
- 可选依赖缺失时，工具不能出现在模型 schema 中。
- 不同平台和任务只应开启相应工具集。
- 外部动态工具和插件不能要求改动核心分发代码。

## 3. 可选方案

### 方案 A：在主循环中手写所有工具 schema 和分发分支

优点是直观；缺点是工具越多，schema、handler、环境检查会分散，容易出现“模型看得到但执行不了”的不一致。

### 方案 B：每个工具文件自带注册声明，中央注册表统一查询和分发

工具模块导入时向注册表提交名称、工具集、schema、handler、check 函数和依赖元数据。主循环只按工具集解析可见工具，再由注册表返回通过可用性检查的 schema，并在执行时统一分发。

## 4. 决策

采用方案 B。

工具能力按两级过滤暴露给模型：

1. **工具集过滤**：先根据入口配置、启用/禁用列表和复合工具集解析出候选工具名。
2. **可用性过滤**：再对候选工具执行工具级或工具集级检查，只返回实际可用 schema。

执行阶段不根据模型提交的任意名称直接反射调用，而是通过注册表查找已注册 handler。未注册工具返回结构化错误。

## 5. 关键机制

- 工具模块导入即注册，注册条目同时包含 schema 与 handler。
- 注册表对同一个检查函数做结果缓存，避免一个工具集内多工具重复执行环境检查。
- schema 返回前强制补齐名称字段。
- 工具执行统一捕获异常并转成 JSON 错误。
- 动态工具可通过注册/注销机制刷新，不破坏主循环分发入口。

## 6. 代价与权衡

- 工具模块导入顺序成为能力发现的一部分，新增工具必须进入发现列表或动态发现机制。
- check 函数必须轻量且无副作用，否则 schema 构建会变慢或引发隐藏状态变化。
- 过度复合的工具集会降低“只暴露必要能力”的收益，需要定期整理工具集边界。

## 7. 架构不变量

- 模型 schema 中出现的工具必须来自已注册条目。
- 已禁用工具集中的工具不得进入 schema。
- 可用性检查失败的工具不得进入 schema。
- 普通工具执行必须走统一分发入口。

## 8. 适用场景 / 不适用场景

适用于多工具、多平台、插件化智能体。不适用于工具数量极少且无可选依赖的脚本型代理，后者用直接函数调用即可。
