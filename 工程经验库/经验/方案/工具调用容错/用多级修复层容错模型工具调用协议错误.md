# 用多级修复层容错模型工具调用协议错误

## 1. 问题

模型工具调用协议要求 assistant 工具调用与 tool 结果严格配对，工具名必须存在，参数必须是合法 JSON，批量调用还要避免重复执行和并发冲突。历史恢复或模型幻觉在任一环节出错，就可能导致 API 拒绝、误执行或会话中断。

## 2. 适用约束

- 多轮工具循环；历史可能来自 DB/压缩/迁移。
- 工具有副作用，不能盲目并发或重复执行。
- 需给模型自修复机会，但有重试上限（工具名/JSON 默认各 3 次）。
- 修复主要作用于 API 消息副本，不 silently 篡改持久化意图。

## 3. 核心思路

三层修复：**请求前**清理配对与编码；**响应后** fuzzy 修工具名 → 校验 → 有限重试；**执行前**截断委托、去重、安全并发。

## 4. 通用结构

- **PreflightSanitizer**：角色白名单、孤立/缺失 tool 配对、Unicode、严格 API 字段。
- **ToolCallValidator**：fuzzy repair、valid_tool_names 校验、JSON parse。
- **RecoveryLoop**：无效 tool/JSON 计数器，每 user turn 重置。
- **BatchGuard**：delegate cap、`(name, args)` 去重、`_should_parallelize_tool_batch`。

## 5. 处理流程

| 阶段 | 层 | 内容 |
|------|-----|------|
| 1 | 请求前 | tool 配对清理 |
| 2 | 请求前 | Unicode 代理清理 |
| 3 | 响应后 | fuzzy 工具名修复 |
| 4 | 响应后 | 无效工具名 → tool 错误 + partial |
| 5 | 响应后 | JSON 校验与空参 `{}` |
| 6 | 响应后 | 截断参数拒绝 + stream 联动 |
| 7 | 执行前 | delegate 截断 + 去重 |
| 8 | 执行前 | 安全并发 / 串行 |
| 9 | 请求前 | 严格 API 剥离扩展字段（可选路径） |

## 6. 异常处理

- 孤立 tool 结果删除；缺失 stub `[结果不可用——请参阅上面的上下文摘要]`。
- 无效工具名 3 次 → `partial=True` 结束回合。
- 截断 JSON → 不执行，partial 返回。
- 非截断无效 JSON 3 次 → 注入 tool 错误，继续循环。
- 工具线程异常 → 单工具错误字符串，不阻断批次。

## 7. 具体语言实现

见各阶段原子卡；主实现在 `run_agent.py`（非 model_tools）。

## 8. 测试点

- 孤立 tool 删除、缺失 stub 补齐。
- fuzzy repair 修正大小写/连字符工具名。
- 无效工具名第 3 次 partial；消息为 tool role 非 user。
- 空参数变 `{}`；截断 JSON 不执行。
- delegate 超上限截断；同名同参只执行一次。
- clarify 批次不并发；重叠 path 不并发。

## 9. 适用场景 / 不适用场景

适用于函数调用智能体、有副作用工具、长历史。不适用于无工具或编排器完全生成调用的系统。

## 10. 风险与反模式

- 把无效工具错误写成 user 消息 → 破坏 role 交替。
- 无效 JSON 无上限重试 → 成本风暴。
- 并发只看数量不看工具名 → 数据竞态。
