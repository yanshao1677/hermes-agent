# 阶段3：工具名模糊修复后再校验

## 触发条件

模型响应含 `tool_calls`，进入工具名校验**之前**，对每个 `tc.function.name` 执行 repair。

## 输入字段

- `tool_name`：模型原始名称。
- `valid_tool_names`：当前可用工具名集合。
- `difflib.get_close_matches(..., n=1, cutoff=0.7)`。

## 判定规则

对不在 `valid_tool_names` 的名称，按序尝试，**命中即返回**：

1. **小写**：`tool_name.lower()` ∈ valid → 返回小写名。
2. **规范化**：`lower.replace("-","_").replace(" ","_")` ∈ valid → 返回规范化名。
3. **模糊匹配**：`get_close_matches(lowered, valid_tool_names, n=1, cutoff=0.7)` 非空 → 返回首个 match。
4. 否则 → 返回 `None`（不修改 tc）。

**应用**

- 若 repair 非空：就地 `tc.function.name = repaired`（可打印修正日志）。
- 然后才进入阶段 4 的 invalid 列表检测。

## 状态读写位置

- 写：当前响应对象上的 `function.name`（内存）。
- 不写 messages 直到阶段 4 决定 append。

## 正常路径

`Web-Search` → repair 为 `web_search` → 阶段 4 校验通过。

## 分支路径

- 名称已 valid → repair 不调用或返回 None，无变化。
- 多个 equidistant fuzzy match → difflib 只取 n=1。

## 失败处理

repair 失败不抛异常，交给阶段 4。

## 幂等性 / 一致性约束

repair 只作用于当前 API 响应，不回写历史。cutoff=0.7 可能误匹配，依赖 valid 集合边界。

## 代码骨架

```python
def repair_tool_name(name, valid_names):
    if name in valid_names:
        return None
    lowered = name.lower()
    if lowered in valid_names:
        return lowered
    normalized = lowered.replace("-", "_").replace(" ", "_")
    if normalized in valid_names:
        return normalized
    matches = get_close_matches(lowered, valid_names, n=1, cutoff=0.7)
    return matches[0] if matches else None
```

## 最小验证清单

- 大小写错误工具名被修复，不触发 invalid 计数。
- 连字符变下划线可修复。
- 完全无关名称 repair 返回 None，进入阶段 4。
