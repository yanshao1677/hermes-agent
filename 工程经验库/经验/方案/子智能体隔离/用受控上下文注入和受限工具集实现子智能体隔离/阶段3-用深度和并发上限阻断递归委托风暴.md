# 阶段3：用深度和并发上限阻断递归委托风暴

## 触发条件

处理委托工具调用，准备创建一个或多个子智能体。

## 输入字段

- 父智能体的 `delegate_depth`。
- `MAX_DEPTH`：深度上限，值为 2。
- `tasks`：批量任务列表，可选。
- `max_concurrent_children`：最大并发子任务数，配置优先，其次环境变量，默认 3。

## 判定规则

1. 当前深度 `>= MAX_DEPTH` 时，拒绝创建子智能体，返回深度上限错误。
2. 创建子智能体后，子深度 = 父深度 + 1。
3. 批量任务数超过最大并发子任务数时，拒绝整个委托请求。
4. 批量任务合法时，用线程池执行，线程池 `max_workers` 等于最大并发子任务数。
5. 最大并发子任务配置不能小于 1；非法配置回退默认值。

## 状态读写位置

- 读：父智能体深度、配置、环境变量。
- 写：子智能体深度字段。
- 不写：父历史中的中间工具轨迹。

## 正常路径

1. 读取父深度。
2. 深度未超限则解析任务列表。
3. 读取最大并发子任务数。
4. 若任务数合法，为每个任务构造子智能体。
5. 单任务直接运行；批任务用线程池并发运行。
6. 汇总子任务结果返回父级。

## 分支路径

- 深度超限 → 返回错误 JSON。
- 任务数超限 → 返回工具错误，不运行任何子任务。
- 配置值非法 → 记录警告并使用默认值。
- 环境变量非法 → 忽略并使用默认值。

## 失败处理

证据不足：单个子任务运行失败时结果结构的完整字段未在本卡展开；本卡确认并发创建前的深度和数量限制。

## 幂等性 / 一致性约束

深度限制和并发限制是资源保护，不保证任务结果幂等。批量请求超过上限时必须整体拒绝，不能悄悄只执行前 N 个，否则父级会误以为所有任务完成。

## 代码骨架

```python
MAX_DEPTH = 2
DEFAULT_MAX_CHILDREN = 3

def delegate(parent, tasks):
    if parent.depth >= MAX_DEPTH:
        return {"error": f"Delegation depth limit reached ({MAX_DEPTH})"}
    max_children = read_max_children(default=DEFAULT_MAX_CHILDREN)
    if len(tasks) > max_children:
        return {"error": f"Too many tasks: {len(tasks)} > {max_children}"}
    children = [build_child(t, parent_depth=parent.depth + 1) for t in tasks]
    with ThreadPoolExecutor(max_workers=max_children) as pool:
        return list(pool.map(run_child, children))
```

## 最小验证清单

- 父深度为 2 时委托直接失败。
- 父深度为 0 时子深度为 1。
- 默认最大并发为 3。
- 传入 4 个任务且上限为 3 时，0 个子任务启动。
- 配置为 0 或非法字符串时，上限回退为至少 1 或默认值。
