# 阶段9：只持久化真实历史并用刷新索引避免重复写入

## 触发条件

一次对话运行准备落盘时触发，包括正常结束、错误退出或提前返回路径。若 `persist_session == False`，则直接跳过。

## 输入字段

- `messages`：真实历史消息列表，不包含 system 请求副本、prefill、临时 user 注入。
- `conversation_history`：本轮传入的旧历史列表或空值。
- `last_flushed_db_idx`：进程内整数游标。
- `session_db`：可选会话数据库。
- `persist_user_message_override` / `persist_user_message_idx`：见阶段 7。

## 判定规则

1. 如果 `persist_session == False`，不保存日志，不写数据库。
2. 保存前先执行阶段 7 的用户消息覆盖。
3. JSON 会话日志保存完整真实历史。
4. SQLite 写入前先 `ensure_session()`（幂等插入）。
5. `start_idx = len(conversation_history) if conversation_history else 0`。
6. `flush_from = max(start_idx, last_flushed_db_idx)`。
7. 只遍历 `messages[flush_from:]` 写入数据库。
8. 写入完成后 `last_flushed_db_idx = len(messages)`。
9. SQLite 失败记录 warning，不中断主流程。

## 状态读写位置

- 读：真实历史、旧历史长度、刷新游标。
- 写：会话日志、SQLite 消息表、`last_flushed_db_idx`。
- 不写：`api_messages`、system/prefill/临时注入。

## 正常路径

1. 检查持久化开关。
2. 应用用户消息覆盖（阶段 7）。
3. 覆盖写 JSON 日志。
4. 计算 `flush_from`，增量 append 消息。
5. 更新游标。

## 分支路径

- 无 `session_db` → 只写 JSON 日志。
- 有旧历史 → 默认跳过 `len(conversation_history)` 以内消息。
- 已部分刷新 → `last_flushed_db_idx` 防重复。
- 压缩后 `conversation_history=None` 且游标为 0 → 新 session 全量写入。
- DB 失败 → warning，继续。

## 失败处理

SQLite 失败不影响响应交付。JSON 日志写入失败的具体分支证据不足，本卡仅确认 SQLite 路径。

## 幂等性 / 一致性约束

同一实例多次 `_persist_session` 不会重复写入已刷新下标之前的消息。跨进程并发写同一 session 需 DB 层额外约束。压缩创建新 session 时必须重置 `last_flushed_db_idx = 0`（阶段 10）。

## 代码骨架

```python
def flush_messages(messages, history, state):
    if not state.persist_session:
        return
    apply_persist_override(messages, state.persist_idx, state.persist_override)
    save_session_log(messages)
    if not state.session_db:
        return
    state.session_db.ensure_session(state.session_id, ...)
    start_idx = len(history) if history else 0
    flush_from = max(start_idx, state.last_flushed_db_idx)
    for msg in messages[flush_from:]:
        state.session_db.append_message(...)
    state.last_flushed_db_idx = len(messages)
```

## 最小验证清单

1. 3 条旧历史 + 2 条新消息 → 只写 2 条。
2. 连续两次 persist 相同 `messages` → 第二次写 0 条。
3. `api_messages` 的 system/prefill 不进入 flush。
4. 压缩后新 session 从 index 0 写入全部压缩后消息。
