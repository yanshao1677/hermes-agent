# 用提供者编排层统一外部记忆预取与同步

## 1. 问题

外部记忆后端（向量库、第三方用户模型 API 等）与本地文件偏好层并存时，若在每个调用点散落 prefetch/sync 逻辑，会出现：工具 schema 冲突、一个 provider 失败阻断另一个、召回内容被当成用户输入持久化、或每轮改动系统提示词前缀。

## 2. 适用约束

- 本地文件偏好层已存在或并行启用。
- 同时最多激活**一个**外部 provider（与内置并存）。
- 外部召回应参与当前任务但不污染真实历史。
- provider 失败不能阻断主对话。

## 3. 核心思路

`MemoryManager` 作为唯一编排点：注册 provider → 聚合 stable system block → 回合前 `prefetch_all` → 围栏注入用户消息副本 → 回合后 `sync_all` + `queue_prefetch_all` → 压缩/写入/委托时触发生命周期钩子。

## 4. 通用结构

- **MemoryProvider (ABC)**：`system_prompt_block`、`prefetch`、`sync_turn`、`queue_prefetch`、工具 schema/handler、生命周期钩子。
- **MemoryManager**：provider 列表、`_has_external` 闸门、工具名路由表。
- **build_memory_context_block**：`<memory-context>` 围栏 + system note + sanitize 剥 tag。

## 5. 处理流程

| 阶段 | 内容 |
|------|------|
| 1 | 注册：builtin 恒接受；第二个 external 拒绝 |
| 2 | prefetch → 围栏 → 追加到 current user API 副本 |
| 3 | final_response 后 sync + queue |
| 4 | on_pre_compress / on_memory_write / on_delegation / shutdown |

## 6. 异常处理

- 单 provider 的 prefetch/sync/hook 异常：log + 跳过，不阻断其它 provider 或主响应。
- 第二个 external 注册：warning + return，不添加。
- 工具名冲突：warning + 忽略后者。

## 7. 具体语言实现

见 `agent/memory_manager.py` 与各阶段原子卡。

## 8. 测试点

- 两 external 同时配置时只有一个生效。
- prefetch 失败时主 API 仍调用。
- prefetch 文本不在 session DB 用户消息中。
- sync 在 final_response 为空时不调用。
- builtin memory write 触发 `on_memory_write` 通知 external。

## 9. 适用场景 / 不适用场景

适用于需对接 Honcho/Mem0 等后端且保留本地偏好的助手。不适用于多后端并行合并、或要求 external 块每轮实时重建进 system prefix 的场景。

## 10. 风险与反模式

- 多个 external 并行 → schema 膨胀与写入冲突。
- prefetch 进 system prompt → 缓存失效。
- 用注入后的 user_message（含 skill）做 sync 查询 → 污染 provider。
