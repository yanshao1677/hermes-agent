# 阶段3：回合结束同步并排队下轮预取

## 触发条件

`run_conversation()` 构建 result 之后、后台审查之前；`_memory_manager` 非空。

## 输入字段

| 字段 | 说明 |
|------|------|
| `original_user_message` | 干净用户输入（非 skill 注入版） |
| `final_response` | 本轮最终助手文本 |
| `session_id` | 可选，传给 provider |

## 判定规则

**调用门槛**

- 仅当 `final_response`  truthy **且** `original_user_message` truthy 时执行。
- 外层 try/except 吞掉异常。

**sync_all(user, assistant, session_id=...)**

- 遍历 providers：`provider.sync_turn(user_content, assistant_content, session_id=...)`。
- 单 provider 失败 → `logger.warning`，继续下一个。

**queue_prefetch_all(user, session_id=...)**

- 遍历 providers：`provider.queue_prefetch(query, session_id=...)`。
- 失败 → debug log，继续。

**时序**

1. 用户已可收到 `final_response`（result 已组装）。
2. sync + queue 执行。
3. 然后才 `_spawn_background_review`（若适用）。

**不用作 sync 输入**

- `user_message`（可能含 skill 注入）——注释明确会膨胀/破坏 provider 查询。

## 状态读写位置

- 写：external provider 后端状态。
- 不写：本地 messages、system prompt。

## 正常路径

1. 主循环产出 final_response。
2. sync 本轮 user/assistant 对。
3. queue 下轮 prefetch 优化。

## 分支路径

- final_response 为空（工具循环未产出文本）→ 跳过 sync/queue。
- manager 为空 → 跳过。
- sync 失败 → 不影响已返回的 result。

## 失败处理

非阻断。无自动 retry。证据不足：各 provider 内部 sync 超时值未统一规定。

## 幂等性 / 一致性约束

编排层不保证 sync 幂等；依赖 provider 实现。queue 为性能优化，非 correctness 必需。

## 代码骨架

```python
def finish_external_memory_turn(manager, original_user_message, final_response, session_id=""):
    if not manager or not final_response or not original_user_message:
        return
    try:
        manager.sync_all(original_user_message, final_response, session_id=session_id)
        manager.queue_prefetch_all(original_user_message, session_id=session_id)
    except Exception:
        pass
```

## 最小验证清单

- final_response=None 时不调用 sync。
- sync 抛异常后 result 仍含 final_response。
- 使用 original 而非 injected user_message 调用 sync。
