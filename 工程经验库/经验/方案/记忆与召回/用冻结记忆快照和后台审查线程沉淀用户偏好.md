# 用冻结记忆快照和后台审查线程沉淀用户偏好

## 1. 问题

智能体需要长期记住用户偏好、环境事实和工作习惯。若每次写记忆后立即改变系统提示词，会导致会话中规则漂移和提示词缓存失效；若完全不自动沉淀，又需要用户手动维护。

本方案只覆盖**层 1 本地文件偏好**（双文件存储 + 冻结快照 + 工具 CRUD + 压缩 flush + 后台审查）。外部 provider 预取/sync 见独立方案《用提供者编排层统一外部记忆预取与同步》；历史情景见《用全文检索和摘要实现历史会话情景召回》。

## 2. 适用约束

- 长期记忆块会被注入系统提示词，必须防注入、限长、去重。
- 当前会话的系统提示词稳定性优先于「写入后立刻可见」。
- 记忆沉淀与压缩前 flush 不应阻塞用户可见响应。
- 情景回忆（任务进度、上次对话细节）不得写入本方案的偏好文件。

## 3. 核心思路

维护**双状态**：加载时渲染并冻结 `_system_prompt_snapshot` 供系统提示词读取；工具读写 **live entries** 并原子落盘。会话内写入不改快照；压缩后 `_invalidate` 触发 `load_from_disk()` 刷新快照。轮次 nudge 驱动主响应交付后的后台审查线程自动沉淀。

## 4. 通用结构

- **MemoryStore**：双 target（`memory` / `user`），char limit，快照 + live。
- **Entry delimiter**：`\n§\n` 分隔条目，支持多行。
- **Threat scanner**：写入前扫描注入/窃密模式与隐形 Unicode。
- **File lock + atomic rename**：并发安全读写。
- **Nudge counters**：`turns_since_memory` 驱动后台审查；memory 工具调用清零。
- **Flush runner**：压缩前仅 memory 工具的单次 LLM 调用，sentinel 清理临时消息。
- **BackgroundReviewer**：daemon 线程，静默子智能体，max_iterations=8。

## 5. 处理流程

| 阶段 | 决断点 | 衔接 |
|------|--------|------|
| 1 | 初始化加载并捕获冻结快照 | → 系统提示词构建读快照 |
| 2 | memory 工具 add/replace/remove | 写 live + 磁盘，快照不变 |
| 3 | 系统提示词读快照 vs 工具响应读 live | 并行一致性感知 |
| 4 | nudge 计数与 memory 调用重置 | → 阶段 7 触发条件 |
| 5 | 压缩前 flush | 必须在阶段 6 之前 |
| 6 | 压缩后失效快照并 reload | 刷新系统提示词 |
| 7 | 主响应后后台审查 | 依赖阶段 4 标志 |

## 6. 异常处理

- 初始化失败：跳过记忆，不破坏智能体启动。
- 超 char limit：拒绝写入，返回当前 usage。
- 注入扫描命中：拒绝写入，返回 blocked 原因。
- flush API 失败：移除 sentinel 消息，不污染历史。
- 后台审查失败：尽力而为，不影响主响应。

## 7. 具体语言实现

见各阶段原子卡的代码骨架；完整 `MemoryStore` 见 `tools/memory_tool.py`。

## 8. 测试点

- 写记忆后 `format_for_system_prompt()` 不变，`add()` 返回 live entries 已更新。
- 重复条目 add 返回 success 但不重复写入。
- replace 多 match 非 identical 时拒绝。
- nudge 达阈值且 final_response 非空时启动审查；中断回合不启动。
- flush 后 sentinel 消息不在 messages 中残留。
- 压缩后快照包含 flush 期间写入的内容。

## 9. 适用场景 / 不适用场景

适用于个人助手、长会话偏好沉淀。不适用于同事务强实时记忆、审计级即时生效、无限增长的 raw 日志存储。

## 10. 风险与反模式

- 把任务进度写进偏好文件——应走情景召回。
- 跳过后台审查的迭代/工具限制——会与主任务争资源。
- 会话中途 reload 快照——破坏前缀缓存。
