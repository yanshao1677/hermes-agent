# 阶段3：系统提示词读快照而工具响应读实时条目

## 触发条件

- **读快照**：构建或恢复 `_cached_system_prompt` 时。
- **读 live**：memory 工具任意 action 成功后返回 JSON。

## 输入字段

| 消费方 | 读取源 | 字段 |
|--------|--------|------|
| `_build_system_prompt()` | `_system_prompt_snapshot` | `memory` / `user` 渲染块 |
| memory 工具响应 | `memory_entries` / `user_entries` | 原始条目列表 + usage |

## 判定规则

1. 系统提示词路径：`format_for_system_prompt(target)` → 仅 `_system_prompt_snapshot.get(target)`；空则跳过该层。
2. 工具响应路径：`_success_response()` → `entries = _entries_for(target)`（live 列表）；`usage = f"{pct}% — {current:,}/{limit:,} chars"`，其中 `current = len(DELIMITER.join(entries))`。
3. 会话中途 add 后：快照块字符数不变；工具 JSON 的 `entries` 和 `usage` 立即反映新条目。
4. 模型若需知晓刚写入内容：必须读 tool result 中的 live `entries`，不能假设 system prompt 已更新。

## 状态读写位置

- 快照：只读（本阶段内）。
- live entries：工具写入路径更新（阶段 2）。

## 正常路径

1. 会话开始：快照进入 `_cached_system_prompt` 一次（或从 session DB 恢复旧 prompt，其中已含旧快照内容）。
2. 模型调用 memory add。
3. API 后续请求的 system 前缀不变。
4. 工具结果展示 100% usage 和新 entries，模型可据此决定 replace/remove。

## 分支路径

- 快照为空（初始无条目）→ 系统提示词无 MEMORY/USER 块；首次 add 后 tool result 有 entries 但 prompt 仍无块直到 reload。
- 续会话从 DB 恢复 system prompt → 即使磁盘已被其它进程更新，本会话仍用恢复的旧块，直到压缩刷新。

## 失败处理

无独立失败路径；不一致是设计行为。若产品要求「写入后立刻进 prompt」，必须走压缩刷新或新会话，不能在本阶段改快照。

## 幂等性 / 一致性约束

**会话级 eventual consistency**：磁盘 = live = 真源；快照 = 会话开始（或上次压缩）视图。后台审查共享 `_memory_store` 写 live，同样不刷新主会话快照。

## 代码骨架

```python
# 系统提示词 consumer
mem_block = store.format_for_system_prompt("memory")  # snapshot only
if mem_block:
    prompt_parts.append(mem_block)

# 工具 consumer
resp = store.add("user", "Prefers terse replies")
# resp["entries"] == live list; format_for_system_prompt("user") unchanged this session
```

## 最小验证清单

- add 前后 `_cached_system_prompt` 字符串相等。
- add 后 tool JSON `entry_count` 增 1。
- 新会话或阶段 6 reload 后 system prompt 含新条目。
