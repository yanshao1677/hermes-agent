# 阶段2：普通工具统一走注册表分发并注入运行时上下文

## 触发条件

模型返回的工具调用不属于 Agent 级状态敏感工具集合。

## 输入字段

- `function_name`：工具名。
- `function_args`：工具参数字典。
- `task_id`：任务隔离标识。
- `tool_call_id`：模型工具调用标识。
- `session_id`：当前会话标识。
- `enabled_tools`：当前实际可用工具名列表，可用于限制沙箱类工具。

## 判定规则

1. 先按工具 schema 对字符串参数做安全类型转换，例如字符串数字转整数、字符串布尔值转布尔值。
2. 非读/搜索类工具执行前通知读循环追踪器，重置连续读取计数。
3. 若工具名属于 Agent 级工具集合，直接返回防漏错误。
4. 执行普通工具前触发工具前置钩子；钩子失败不阻断。
5. 对需要知道可用工具范围的工具，优先使用调用方传入的 `enabled_tools`，避免读取可能被子智能体覆盖的进程全局列表。
6. 其它工具通过注册表按名称分发，并传入通用运行时上下文。
7. 工具执行后触发后置钩子；钩子失败不阻断。

## 状态读写位置

- 读：注册表 schema、当前工具名集合、运行时上下文字段。
- 写：无持久写入；工具自身可能有副作用。
- 进程全局最近工具名列表只作为兜底，不应优先于调用方显式传入的可用工具列表。

## 正常路径

1. 参数按 schema 做类型转换。
2. 触发非读工具的读取计数重置。
3. 防漏检查 Agent 级工具。
4. 调用前置钩子。
5. 调用注册表分发工具。
6. 调用后置钩子。
7. 返回工具结果字符串。

## 分支路径

- schema 不存在 → 不做参数类型转换。
- 参数转换失败 → 保留原值。
- 前置或后置钩子异常 → 跳过钩子错误，继续。
- 未知工具 → 注册表返回结构化错误。
- 工具处理器异常 → 注册表返回结构化执行失败错误。

## 失败处理

普通工具处理器异常不会向上抛出破坏主循环，而是包装为 JSON 错误。证据不足：插件钩子失败是否有指标记录未深读；当前证据只显示异常被吞掉。

## 幂等性 / 一致性约束

通用分发层不保证工具副作用幂等；它只保证协议层统一返回字符串结果。需要幂等的工具必须在自身处理器内实现。

## 代码骨架

```python
def handle_function_call(name, args, context):
    args = coerce_args_from_schema(name, args)
    if name in AGENT_LOOP_TOOLS:
        return json.dumps({"error": f"{name} must be handled by the agent loop"})
    try:
        invoke_hook("pre_tool_call", tool_name=name, args=args, session_id=context.session_id)
    except Exception:
        pass
    result = registry.dispatch(name, args, task_id=context.task_id, enabled_tools=context.enabled_tools)
    try:
        invoke_hook("post_tool_call", tool_name=name, args=args, result=result, session_id=context.session_id)
    except Exception:
        pass
    return result
```

## 最小验证清单

- 字符串 `"3"` 在 schema 期望整数时被转换为整数。
- Agent 级工具误入通用分发时返回错误 JSON。
- 未知工具名返回 `Unknown tool` 错误。
- 工具处理器抛异常时返回 `Tool execution failed` 错误。
- 显式传入 `enabled_tools` 时，不依赖全局最近工具名列表。
